# LawLLM2 간단 소개서 (입문자용)

이 문서는 **인공지능 배경지식이 거의 없는 분**이 `LawLLM2` 프로젝트를 빠르게 이해하도록 만든 안내서입니다.

---

## 1) 이 프로젝트는 무엇을 하나요?

`LawLLM2`는 건축 법규 문서를 바탕으로 질문에 답해주는 **간단한 RAG 기반 질의응답 도구**입니다.

- 질문 예시: "일조권 관련 핵심 조문은 뭐야?"
- 동작 방식: 문서를 먼저 찾고(검색), 찾은 문서 내용을 근거로 LLM이 답변
- 목표: "모르는 내용을 꾸며내는 답변"을 줄이고, **근거 기반 답변**을 하도록 유도

---

## 2) RAG가 무엇인가요?

RAG는 `Retrieval-Augmented Generation`의 약자입니다.

쉽게 말하면:
1. **Retrieval(검색)**: 질문과 관련된 문서 조각을 먼저 찾고
2. **Augmented(보강)**: 찾은 내용을 질문과 함께 LLM에게 전달한 뒤
3. **Generation(생성)**: LLM이 그 근거를 참고해 답변을 생성

즉, LLM의 "기억"만 믿지 않고, **내 데이터에서 근거를 찾아 답변**하게 만드는 방법입니다.

---

## 3) 현재 프로젝트 구조

- `main.py`  
  핵심 실행 파일(인덱싱, 검색, 답변 생성)
- `data/`  
  법규 `.md` 원문 데이터 폴더
- `index_cache.pkl`  
  검색을 빠르게 하기 위한 인덱스 캐시 파일
- `evaluate_readiness.py`  
  표준 지표 계산을 위한 내부 평가 모듈
- `run_selection_eval.py`  
  실험 실행 진입점(게이트+우선순위 규칙으로 최종 인덱싱 방식 선택)
- `eval_queries.json`  
  평가용 질의셋(질문 + 기대 근거)

---

## 4) 현재 검색(인덱싱) 방식은?

이 프로젝트는 현재 **단일 + 하이브리드 검색**을 지원합니다.

- `tfidf`: 키워드 기반 희소 벡터 검색
- `bm25`: 전통적인 정보검색 점수 방식
- `dense`: 저차원 밀집 벡터 검색
- `hybrid_tfidf_bm25`: TF-IDF + BM25
- `hybrid_bm25_dense`: BM25 + Dense
- `hybrid_tfidf_dense`: TF-IDF + Dense

또한 질문에 `제86조`, `일조`, `채광` 같은 신호가 있으면 점수 보정을 추가해 관련 청크를 올리도록 되어 있습니다.

---

## 5) 실행 흐름 (한 줄 요약)

`문서 읽기 -> 청크 분할 -> 인덱스 생성 -> 질문 검색 -> 근거를 프롬프트에 넣어 LLM 호출 -> 답변 출력`

### 주요 명령어

```bash
python main.py index
python main.py ask "일조권 관련 조문 알려줘" --retrieval-mode tfidf --top-k 6
python run_selection_eval.py
```

---

## 6) 답변이 길거나 부정확할 때 왜 그런가요?

주요 원인은 보통 2가지입니다.

1. **검색 실패**: 질문과 맞는 조문 청크를 못 찾음
2. **문서 노이즈**: 본문 외 부칙/개정 이력/표가 섞여 관련도가 흐려짐

그래서 답변 품질은 LLM 성능만이 아니라, **검색 품질(인덱싱/청킹/질의셋)**에 크게 좌우됩니다.

---

## 7) 어떻게 품질을 검증하나요?

검색 방식 우열은 체감이 아니라 **질의셋 기반 표준 지표**로 비교합니다.

- `Recall@k`: 정답 근거가 상위 k개 안에 있나?
- `Precision@k`: 상위 k개 근거 중 유효 근거 비율은?
- `MRR`: 정답 근거가 몇 번째로 나왔나?
- `Readiness Accuracy`: 판단 가능/불가 분류를 맞췄나?

같은 질의셋으로 여러 검색 모드를 동일 조건에서 돌려 공정 비교합니다.

---

## 8) 현재 프로젝트의 장점/한계

### 장점
- 구조가 단순해서 이해와 수정이 쉬움
- 법규 QA 입문용으로 빠르게 실험 가능
- 검색 모드 비교와 평가 루프가 있음

### 한계
- 문서 전처리(조문 단위 분리) 미흡 시 검색 정확도 하락
- 대규모 데이터/고도화된 의미검색에는 한계
- LLM이 근거를 잘못 요약할 가능성 존재

---

## 9) 다음 개선 추천 (우선순위)

1. **조문 단위 청킹** (`제N조` 기준 분할)
2. 평가 질의셋 확대(20 -> 50+) 및 정답 기준 정교화
3. 필요 시 임베딩 기반 검색을 추가해 하이브리드 고도화

---

## 10) 용어 미니 사전

- **LLM**: 큰 언어 모델(질문에 답하는 AI 모델)
- **RAG**: 검색 + 생성 결합 방식
- **인덱싱**: 검색을 빠르게 하기 위한 사전 준비 작업
- **청크(Chunk)**: 문서를 잘게 나눈 조각
- **근거(Context)**: 답변 생성 시 참고하는 문서 조각
